# Short Report: Adaptive RAG Inference System

## 1. What Worked

### Adaptive K Selection
The query complexity analyzer effectively distinguishes between simple factual queries ("What is TCP?") and complex analytical queries ("Compare supervised and unsupervised learning"). By varying K from 3 to 10, simple queries get faster responses while complex queries get more comprehensive context.

### Hybrid Retrieval
Combining FAISS vector search with BM25 keyword search via Reciprocal Rank Fusion proved highly effective. Vector search handles semantic similarity (matching "automobile" to "car"), while BM25 catches exact technical terms that embeddings might dilute. RRF elegantly combines these without needing score normalization.

### Cross-Encoder Re-Ranking
The `ms-marco-MiniLM-L-6-v2` cross-encoder significantly improved result quality for complex queries. By scoring query-document pairs jointly (rather than independently like bi-encoders), it better assesses fine-grained relevance. The adaptive layer's decision to skip re-ranking for simple queries keeps latency low when it's not needed.

### Semantic Caching
The cosine-similarity based cache provides near-instant responses for repeated or similar queries. A threshold of 0.92 balances between serving relevant cached results and avoiding stale answers for genuinely different queries.

## 2. What Didn't Work / Challenges

### Quality Proxy Limitations
Without ground-truth labels, the quality proxy (keyword overlap, answer length, retrieval scores) is a rough approximation. Long, generic answers score well even if they don't precisely answer the question. A better approach would use LLM-as-judge evaluation, but that adds latency and API costs.

### Feedback Loop Responsiveness
The rule-based feedback loop can oscillate between states — increasing K due to low quality, then decreasing due to high latency, creating a feedback cycle. The rolling window (20 queries) smooths this, but aggressive settings could cause instability.

### Query Decomposition
Simple heuristic-based decomposition (splitting on conjunctions and question marks) works for obvious cases but misses subtle multi-part queries. An LLM-based decomposer would be more accurate but adds latency.

## 3. How the System Adapts

### Inference-Time Adaptation
```
Query → Complexity Score → Parameter Selection → Retrieval → Generation
         (0.0 - 1.0)       (K, strategy,          (FAISS +    (Gemini /
                             reranker)              BM25)       mock)
```

**Example adaptation sequence:**

| Query | Complexity | K | Strategy | Re-ranker | Rationale |
|-------|------------|---|----------|-----------|-----------|
| "What is SQL?" | LOW (0.15) | 3 | vector_only | OFF | Short, simple → fast path |
| "Explain transformers in NLP" | MEDIUM (0.45) | 5 | hybrid | ON | Technical explanation → balanced |
| "Compare SQL vs NoSQL databases for different workloads" | HIGH (0.72) | 10 | hybrid | ON | Comparison + multi-faceted → thorough |

### Cross-Session Learning
The feedback history is persisted to JSON, so the system "remembers" past performance across sessions. If a certain type of query consistently produces poor results with K=3, the feedback loop will push K higher for similar future queries.

### Latency-Aware Adjustments
When the system detects latency exceeding the SLA threshold (8s), it automatically:
1. Reduces K by 2
2. Disables the cross-encoder re-ranker
3. Switches to vector-only search

This degradation is graceful — the system trades some quality for responsiveness.

## 4. Key Metrics

Performance measurements are generated by the benchmark suite (`python main.py benchmark`). Expected patterns:

- **Simple queries**: ~0.5-1.5s total latency, K=3, vector-only
- **Medium queries**: ~1.5-3.0s total latency, K=5, hybrid + re-ranker
- **Complex queries**: ~2.5-5.0s total latency, K=10, hybrid + re-ranker + decomposition
- **Retrieval typically accounts for 20-40% of total latency**
- **Generation (LLM) accounts for 60-80% of total latency**

## 5. Future Improvements

1. **LLM-as-Judge**: Use the LLM to evaluate answer quality instead of heuristics
2. **Learned Adaptive Layer**: Train a small model to predict optimal K from query features
3. **ANN Tuning**: Experiment with FAISS IVF/HNSW indexes for larger document collections
4. **Model Routing**: Use a smaller/faster model for simple queries and a larger model for complex ones
5. **Streaming Generation**: Stream LLM responses for better perceived latency
